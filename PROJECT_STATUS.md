## Ampliação: Serviços (pós Feature 000)

Adiciona ao módulo `catalog` (Flyway `V007`) os campos cor, ordem de exibição, exigência de confirmação e inativação — pedidos explicitamente do escopo funcional completo (seção "Serviços"), fora da fatia mínima da Feature 000.

- **Cor**: opcional, validada no formato hexadecimal `#RRGGBB` (`@Pattern`); `null` permanece válido (campo não obrigatório).
- **Ordem de exibição**: opcional na requisição; quando omitida, é calculada automaticamente como `max(ordem existente na organização) + 1` (`ServiceRepository.nextDisplayOrder`), então novos serviços aparecem no fim da lista por padrão. Lista ordenada por `display_order` e depois nome.
- **Exige confirmação**: booleano, `false` por padrão.
- **Inativação**: uma ação real (`POST /api/v1/catalog/services/{id}/deactivate`), não só uma coluna passiva — segue a regra do AGENTS.md de nunca apagar serviço já usado, só inativar para preservar histórico. `GET /services` continua retornando ativos *e* inativos (com a flag `active`), já que o histórico precisa continuar visível; o módulo `scheduling` não foi alterado (permanece podendo resolver nome de serviços inativos para agendamentos antigos). A página "Agenda" filtra a lista para mostrar só serviços ativos no dropdown de novo agendamento — um filtro só no frontend, sem tocar no backend de `scheduling`.
- Sem reativação nem edição de serviço — não foi pedido nesta rodada.

**Bug real encontrado e corrigido**: o `CreateServiceRequest` (Java record) tinha um construtor auxiliar de 2 argumentos para compatibilidade com os testes já existentes, e o campo opcional `requiresConfirmation` era um `boolean` primitivo. Ao enviar um JSON de verdade com só os campos obrigatórios (sem passar pelo construtor Java, que sempre preenche todos os campos do record antes de serializar), o Jackson tentava mapear o campo ausente como `null` para um `boolean` primitivo e falhava com `Cannot map 'null' into type 'boolean'` — um 400 sem nenhuma mensagem útil. Corrigido trocando para `Boolean` (boxed, aceita `null`) com um método `requiresConfirmationOrDefault()` resolvendo o padrão `false`. Esse bug só aparece com JSON parcial de verdade — testes que serializam o próprio record Java (`objectMapper.writeValueAsString(new CreateServiceRequest(...))`) nunca o pegam, porque sempre produzem um JSON completo. Adicionado teste de regressão (`acceptsRawJsonPayloadWithOnlyTheRequiredFields`) que envia uma string JSON literal com só os campos obrigatórios. **Lição para requests futuras**: campos opcionais em records usados como corpo de requisição devem ser tipos boxed (`Boolean`, `Integer`), nunca primitivos.

Painel: formulário de serviço ganhou seletor de cor, ordem de exibição (opcional) e checkbox "Exige confirmação"; lista mostra a cor como um indicador visual, tags de "Exige confirmação"/"Inativo", e botão "Inativar" por serviço ativo.

Validado via curl contra Postgres real (a extensão do Chrome ficou instável nesta sessão — aba travada e eventos de clique não registrando mesmo em aba nova; ver ressalva abaixo): cadastro com cor/ordem/confirmação, ordem automática incrementando corretamente, inativação registrando auditoria (`SERVICE_DEACTIVATED`) e mantendo o serviço na listagem com `active: false`.

**Ressalva**: a validação visual no navegador (Claude in Chrome) não foi possível nesta sessão por instabilidade da extensão (erro "Browser extension is not connected", aba sem resposta a scripts). A validação funcional foi feita via curl contra a API real e via os testes automatizados (backend e frontend), mas o fluxo específico desta ampliação não foi clicado manualmente na UI.

## Remarcação e cancelamento de agendamentos (pós Feature 000)

Adiciona ao módulo `scheduling` (Flyway `V008`) a capacidade de remarcar e cancelar agendamentos — a Feature 000 só cobria criação e listagem.

- **Status**: novo campo `status` (`SCHEDULED`/`CANCELLED`) no `Appointment`. A constraint `appointments_no_overlap` (V005) foi recriada como **EXCLUDE parcial** (`WHERE status = 'SCHEDULED'`): sem isso, cancelar um agendamento nunca liberaria o horário, já que a constraint original considerava todas as linhas, inclusive as canceladas. Validado com teste no nível de banco (`AppointmentOverlapConstraintTest.cancellingAnAppointmentFreesUpTheSlotAtTheDatabaseLevel`) e ponta a ponta via API.
- **Remarcar** (`POST /api/v1/appointments/{id}/reschedule`): atualiza `start_at`/`end_at` do mesmo registro (mesmo id, não cria um novo agendamento); revalida sobreposição excluindo o próprio registro (`AppointmentRepository.existsOverlappingExcluding`); não permite remarcar um agendamento já cancelado (`InvalidAppointmentStateException`, 400).
- **Cancelar** (`POST /api/v1/appointments/{id}/cancel`): exige `reason` (`@NotBlank`) para manter o histórico útil; marca `status = CANCELLED` e grava `cancellation_reason`; não permite cancelar duas vezes (400). Nunca apaga a linha — preserva histórico conforme AGENTS.md.
- **Auditoria com metadata**: até aqui o `AuditRecorder` só gravava quem/quando/o quê, sem detalhe (a coluna `metadata JSONB` da V001 nunca tinha sido mapeada, por risco não testado de Hibernate 7 + Jackson 3 — ver nota da etapa 3). Resolvido nesta rodada: `AuditLog.metadata` mapeado como `Map<String, String>` com `@JdbcTypeCode(SqlTypes.JSON)`, validado com Postgres real (Testcontainers). `APPOINTMENT_RESCHEDULED` grava horário anterior e novo; `APPOINTMENT_CANCELLED` grava o motivo — como pedido no AGENTS.md ("quem remarcou; data e horário anteriores; data e horário novos; quem cancelou; motivo do cancelamento"). O contrato `AuditRecorder` ganhou uma sobrecarga com `metadata`; a sobrecarga antiga (sem metadata) continua existindo para as chamadas que não precisam dela.
- Sem lista de espera, confirmação manual ou os demais status do ciclo completo (chegou, em atendimento, realizado, não compareceu) — fora do escopo pedido nesta rodada.

Painel: cada linha de agendamento ativo ganhou botões "Remarcar" e "Cancelar", cada um abrindo um mini-formulário inline na própria linha (sem modal). Remarcar preserva a duração original do agendamento (calculada a partir do `startAt`/`endAt` atuais, não depende de re-selecionar o serviço). Agendamentos cancelados continuam na lista (histórico), com estilo tracejado e o motivo exibido em vermelho.

Validado ponta a ponta: suíte completa do backend (`./mvnw clean verify`, 46 testes, incluindo o `ArchitectureTest` de módulos), suíte do frontend (`ng test`, todos os specs), curl contra API + Postgres reais (criar → remarcar → conflito bloqueado → cancelar com motivo → horário liberado → cancelamento duplo rejeitado → auditoria com os dados corretos), e desta vez também na UI real no navegador (login → Agenda → remarcar → cancelar), sem a instabilidade do Claude in Chrome que tinha bloqueado a validação visual da ampliação de Serviços.

## Horários de funcionamento e bloqueios (pós Feature 000)

Novo módulo `availability` (já previsto no AGENTS.md, com scaffold vazio criado na Fase 1.1) com duas capacidades: horário de funcionamento semanal e bloqueios pontuais. Ambos passam a ser validados na criação e remarcação de agendamentos (`scheduling`), reaproveitando a infraestrutura já existente de checagem de conflito.

- **Horário de funcionamento** (Flyway `V009`, tabela `business_hours`): um intervalo por dia da semana (`day_of_week` + `start_time`/`end_time`, únicos por organização). `PUT /api/v1/availability/business-hours` **substitui inteiramente** a configuração anterior a cada chamada (semântica de "salvar a semana toda"), em vez de criar/editar/remover dias individualmente — mais simples e sem risco de estado parcial inconsistente. **Comportamento padrão preservado**: se a organização nunca configurou horários (tabela vazia), não há nenhuma restrição — o comportamento é idêntico ao que existia antes deste recurso. Só passa a haver restrição depois que a organização configura ao menos um dia; dias não configurados nesse ponto em diante contam como fechados.
- **Bloqueios** (Flyway `V009`, tabela `blocks`): período livre (`start_at`/`end_at` + `reason` obrigatório) em que a organização não atende (feriado, compromisso pessoal, etc.). `POST/GET /api/v1/availability/blocks` e `DELETE /api/v1/availability/blocks/{id}`. Remover um bloqueio é exclusão real (não há "histórico de negócio" a preservar aqui, diferente de cliente/serviço/agendamento), mas a remoção é auditada (`BLOCK_REMOVED`, com motivo e horário nos metadata) antes de apagar a linha.
- **Validação em `scheduling`**: `AppointmentScheduler.create()`/`.reschedule()` agora checam, nesta ordem, (1) se o horário está dentro do funcionamento configurado (`OutsideBusinessHoursException`, 400) e (2) se não sobrepõe um bloqueio (`BlockedTimeException`, 409, código `blocked_time` — distinto do conflito entre agendamentos). Novo contrato público mínimo `availability.AvailabilityCheck` (implementado em `availability.infrastructure`), no mesmo padrão de `ServiceLookup`/`ClientLookup`. `organizations.CurrentOrganization` ganhou o campo `timezone` (já existia na tabela `organizations` desde a V001, nunca tinha sido exposto) para converter o horário do agendamento (`Instant`, UTC) para o dia da semana e hora local da organização antes de comparar com `business_hours`.
- **Limitação conhecida e aceita nesta rodada**: a proteção contra sobreposição agendamento-vs-agendamento tem uma constraint `EXCLUDE` no PostgreSQL como rede de segurança final contra concorrência (ADR 0006); agendamento-vs-bloqueio só tem checagem na aplicação, sem equivalente no banco (uma exclusão cruzando duas tabelas não é direta em Postgres). Dado o volume de uso esperado (uma secretária por organização, não um sistema de alta concorrência), o risco de duas requisições colidindo exatamente nesse instante é considerado aceitável por ora; documentado aqui para revisão futura caso o padrão de uso mude.
- Sem exceções pontuais ao horário semanal (ex.: "só essa sexta abre até mais tarde") — isso é resolvido criando um bloqueio ou não, não uma feature separada nesta rodada. Sem UI de calendário para os bloqueios (aparecem em lista simples, junto do horário de funcionamento, na página "Configurações").

Painel: página "Configurações" ganhou duas seções reais — grade dos 7 dias da semana (marcar/desmarcar + horário de início/fim) com botão "Salvar horários", e uma lista de bloqueios com formulário de criação (início, fim, motivo) e botão "Remover" por item. A página "Agenda" ganhou mensagens de erro específicas por código (`blocked_time`, `invalid_appointment`) em vez do texto genérico anterior — encontrado durante a validação manual no navegador: a UI mostrava "Não foi possível criar o agendamento" tanto para um horário bloqueado quanto para qualquer outro erro, quando a API já retornava uma mensagem específica.

Validado com Postgres real: suíte completa do backend (`./mvnw clean verify`), incluindo `ArchitectureTest` (o novo módulo `availability` não viola limites do Spring Modulith) e testes dedicados de agendamento fora do horário configurado, dentro do horário configurado, sobreposição com bloqueio na criação e na remarcação; suíte do frontend (`ng test`); validação manual via curl; e validação na UI real no navegador (configurar horário de sábado, criar bloqueio, tentar agendar dentro do bloqueio e ver a mensagem "Este horário está bloqueado.").

## Ciclo completo de status do agendamento (pós Feature 000)

Fecha a Fatia 2 do roadmap ("operação diária"): além de agendado/cancelado, o agendamento agora percorre confirmado, chegou, em atendimento e realizado, além de falta — exatamente o que a especificação funcional pedia em "confirmar manualmente no sistema" e "registrar chegada, realização e falta" (Flyway `V010`).

- **Novos status**: `CONFIRMED`, `ARRIVED`, `IN_PROGRESS`, `DONE`, `NO_SHOW`, além dos já existentes `SCHEDULED`/`CANCELLED`.
- **Transições e regras** (todas em `Appointment`, cada uma validando o estado atual e lançando `InvalidAppointmentStateException` com mensagem específica caso inválida):
  - `confirmar`: só a partir de `SCHEDULED`.
  - `registrar chegada`: a partir de `SCHEDULED` ou `CONFIRMED`.
  - `iniciar atendimento`: só a partir de `ARRIVED`.
  - `concluir` (realizado): a partir de `ARRIVED` ou `IN_PROGRESS`.
  - `registrar falta`: a partir de `SCHEDULED` ou `CONFIRMED` (não faz sentido depois que a cliente já chegou).
  - `cancelar`: passou a ser bloqueado se o agendamento já está `DONE` ou `NO_SHOW` (antes só bloqueava se já `CANCELLED`).
  - `remarcar`: passou a exigir `SCHEDULED` ou `CONFIRMED` (antes aceitava qualquer coisa que não fosse `CANCELLED`; agora também bloqueia a partir de `ARRIVED` em diante, já que não faz sentido remarcar um atendimento que já começou).
- **Conflito de horário**: a constraint `EXCLUDE` do PostgreSQL (V005/V008, ADR 0006) e a checagem da aplicação foram generalizadas de "`WHERE status = 'SCHEDULED'`" para "`WHERE status NOT IN ('CANCELLED', 'NO_SHOW')`" — ou seja, `CONFIRMED`/`ARRIVED`/`IN_PROGRESS`/`DONE` continuam ocupando o horário (fazia sentido antes só considerar `SCHEDULED`, mas agora que existem mais status "ativos" todos eles precisam contar); só cancelamento e falta liberam o horário para um novo agendamento.
- **"Próximo atendimento" do dashboard** também foi generalizado: antes só considerava `SCHEDULED`, agora considera qualquer status que não seja `CANCELLED`, `NO_SHOW` ou `DONE` (ou seja, inclui `CONFIRMED`/`ARRIVED`/`IN_PROGRESS`).
- **Endpoints novos**: `POST /api/v1/appointments/{id}/confirm`, `/arrive`, `/start`, `/complete`, `/no-show` — todos sem corpo, seguindo o padrão já usado em `/cancel`/`/reschedule`, cada um auditado (`APPOINTMENT_CONFIRMED`, `APPOINTMENT_ARRIVED`, `APPOINTMENT_STARTED`, `APPOINTMENT_COMPLETED`, `APPOINTMENT_NO_SHOW`).
- **Resumo semanal do dashboard** ampliado de 2 para 4 contadores: `scheduledCount` (ainda em andamento: agendado/confirmado/chegou/em atendimento), `completedCount` (realizado), `cancelledCount`, `noShowCount`.

Painel: a página "Agenda" ganhou botões contextuais por linha — só aparecem as ações válidas para o status atual (ex.: "Iniciar atendimento" só aparece depois de "Registrar chegada"; "Remarcar"/"Cancelar" somem quando o agendamento já foi concluído/cancelado/com falta). Cada status ganhou uma etiqueta visual (Confirmado, Chegou, Em atendimento, Realizado, Não compareceu). O Dashboard também exibe essas etiquetas na lista "Agenda de hoje" e o card "Resumo da semana" ganhou mais dois números (realizados, faltas).

Validado com Postgres real: suíte completa do backend (`./mvnw clean verify`), incluindo `ArchitectureTest` e testes cobrindo cada transição de status, as transições inválidas (ex.: iniciar atendimento sem chegada registrada, remarcar um agendamento com chegada registrada, cancelar um já realizado) e a liberação do horário após falta; suíte do frontend (`ng test`) cobrindo a exibição contextual dos botões e o fluxo confirmar → chegar → iniciar → concluir; e validação manual completa na UI real do navegador (confirmar → registrar chegada → concluir refletido corretamente na Agenda e no Dashboard).

## Recorrência semanal e quinzenal (pós Feature 000)

Abre a Fatia 3 do roadmap ("recorrência e histórico") na parte que ainda faltava — histórico do cliente e histórico de alterações já estavam prontos (ver seções abaixo); esta rodada cobre "recorrência semanal ou quinzenal" do escopo funcional (seção "Agenda"). Escopo deliberadamente contido, como já estava registrado como plano na seção "Histórico do cliente": cada ocorrência é um `Appointment` comum e independente, agrupado por `series_id` (Flyway `V011`, coluna nullable, sem FK — não há tabela de série própria). Editar/cancelar a série inteira de uma vez fica para uma rodada futura, se pedido; por ora cada ocorrência já pode ser remarcada/cancelada/confirmada individualmente através dos endpoints existentes, sem nenhuma mudança neles.

- **Criação**: `POST /api/v1/appointments/recurring` (`CreateRecurringAppointmentRequest`: cliente, serviço, primeiro horário, `frequency` — `WEEKLY`/`BIWEEKLY`, novo enum `RecurrenceFrequency` com o intervalo em dias — e `occurrenceCount`, limitado a 2–52 por `@Min`/`@Max`). Gera as ocorrências somando o intervalo repetidamente ao primeiro horário (mesma duração em todas).
- **Atomicidade**: toda a série é criada em uma única transação; se qualquer ocorrência conflitar (com outro agendamento, bloqueio, ou cair fora do horário de funcionamento), a exceção correspondente já existente (`AppointmentConflictException`/`BlockedTimeException`/`OutsideBusinessHoursException`) propaga e o `@Transactional` desfaz tudo — nenhuma ocorrência parcial fica salva. As ocorrências da mesma série nunca conflitam entre si por construção (mesmo horário do dia, espaçadas por 7 ou 14 dias — sempre mais que a duração de qualquer atendimento razoável).
- **Reaproveitamento**: a lógica de checar disponibilidade + sobreposição + salvar + auditar, que já existia em `create()`, foi extraída para um método privado (`createAndSaveAppointment`) compartilhado entre criação única e em série — evita duplicar a mesma sequência de validações.
- **Auditoria por ocorrência**: cada ocorrência grava seu próprio `APPOINTMENT_CREATED` (mesmo padrão da criação avulsa), com `seriesId` e `occurrenceIndex` no metadata para rastreabilidade — sem um evento de série separado, para não precisar de um tipo novo só para isso.
- `AppointmentSummary` ganhou o campo `seriesId` (nulo para agendamentos avulsos) — usado pelo painel para mostrar a etiqueta "Recorrente".

Painel: o formulário de novo agendamento ganhou um checkbox "Repetir agendamento" que revela frequência (Semanal/Quinzenal) e quantidade de ocorrências; ao criar uma série, todas as ocorrências retornadas entram na lista de uma vez, cada uma com a etiqueta "Recorrente".

Validado com Postgres real: suíte completa do backend (`./mvnw clean verify`), incluindo os quatro testes novos de série (intervalo semanal, intervalo quinzenal, rollback atômico quando uma ocorrência no meio da série conflita, limites de quantidade de ocorrências); suíte do frontend (`ng test`, 36 testes incluindo o novo fluxo de criação recorrente).

## Revisão de qualidade de código (pós Feature 000)

Pausa de manutenção pedida explicitamente ("reavalie com calma") depois de várias rodadas seguidas de funcionalidade nova, cobrindo seis frentes: código morto, duplicação, testes faltantes, magic numbers, módulos grandes demais — com a regra explícita de nunca mudar comportamento sem teste cobrindo a mudança.

- **Código morto**: nenhum encontrado (sem `TODO`/`FIXME`/`XXX` no backend ou frontend, sem métodos ou classes não referenciados).
- **Duplicação**: os seis testes de controller com sessão autenticada (`ClientControllerTest`, `AppointmentControllerTest`, `ServiceControllerTest`, `AvailabilityControllerTest`, `AuditTrailControllerTest`, `DashboardControllerTest`) repetiam, quase sem variação, a lógica de criar usuária + organização + login + obter cookie CSRF (cada um com seu próprio `record AuthenticatedSession`/`Organization` e métodos privados idênticos). Extraído para `br.com.agendaplatform.support.IntegrationTestSupport` (classe utilitária estática, parâmetros explícitos — sem estado compartilhado entre testes). Deliberadamente **não** tocado: o container Testcontainers e `@DynamicPropertySource` continuam duplicados por classe, porque uma base class com container estático mudaria o isolamento "um Postgres por classe de teste" — risco maior que o benefício de deduplicar.

- **Módulos grandes demais**: `AppointmentScheduler` (261 linhas) é a maior classe do backend, por uma margem real sobre a segunda (`Appointment`, 167 linhas) — mas todas as suas responsabilidades (criar, remarcar, cancelar, as 5 transições de status, 3 consultas) giram em torno do mesmo agregado e já compartilham um único mapeador (`toSummary`). No frontend, `agenda-page.component.ts` (246 linhas, maior arquivo do painel) foi avaliado com a mesma conclusão na rodada anterior.

Validado com Postgres real: suíte completa do backend (`./mvnw clean verify`) rodando sem nenhuma mudança de comportamento — os seis arquivos de teste refatorados passam exatamente como antes da extração, mais os 58 testes novos (52 + 6) e o teste adicional de `Block`.

## Lista de espera (Fatia 4 do roadmap)

Fecha a Fatia 4 ("lista de espera") inteira nesta rodada: cadastro, compatibilidade com vagas, conversão em agendamento e indicadores básicos — os quatro itens do roadmap.

- **Novo módulo `waitlist`** (Flyway `V012`, tabela `waitlist_entries`): cliente, serviço, período preferido (`preferredStartDate`/`preferredEndDate`), horário preferido (`preferredStartTime`/`preferredEndTime`), prioridade (`LOW`/`NORMAL`/`HIGH`) e validade (`expiresAt`) — exatamente os campos do escopo funcional ("cliente e serviço; período e horário preferidos; prioridade e validade; situação").
- **Situação**: `WAITING`/`CONVERTED`/`CANCELLED` persistidos; `EXPIRED` é **calculado em tempo de leitura** (comparando `expiresAt` com o relógio atual — mesmo padrão testável de `Clock` já usado no dashboard), nunca gravado no banco — evita depender de um job agendado para manter isso em dia. Cancelar um registro cujo prazo já passou continua permitido (ainda é `WAITING` no banco); converter, não (`WaitlistEntryExpiredException`).
- **Compatibilidade com vagas**: `GET /api/v1/waitlist/compatible?serviceId=&startAt=&endAt=` retorna os registros `WAITING` (não expirados) do mesmo serviço cujo período e horário preferidos cobrem o horário candidato — a checagem de horário converte o instante para o fuso da organização antes de comparar, mesmo padrão já usado em `availability` para horário de funcionamento. Pensado para uso manual: quando surge uma vaga (ex.: um cancelamento), a secretária consulta este endpoint para ver quem poderia ser chamada — sem nenhuma automação de notificação (fora do escopo, WhatsApp é exclusão definitiva).
- **Conversão em agendamento**: `POST /api/v1/waitlist/{id}/convert` (corpo: `startAt`/`endAt` escolhidos manualmente) cria um agendamento de verdade e marca o registro como `CONVERTED`, guardando o `appointmentId`. Novo contrato público mínimo `scheduling.AppointmentBooking` (implementado por `AppointmentScheduler`, que já implementa `AppointmentOverview`) permite ao módulo `waitlist` criar o agendamento reaproveitando toda a validação de disponibilidade já existente (horário de funcionamento, bloqueios, sobreposição) sem acessar as classes internas do `scheduling`. Como as exceções específicas desse fluxo (conflito, bloqueio etc.) pertencem a `scheduling.domain` — interno ao módulo, `WaitlistController` não pode importá-las — a falha da conversão é capturada por um handler genérico de `RuntimeException` mapeado para 409, repassando a mensagem original (já em português, adequada para exibição). Simplificação registrada aqui: perde-se a granularidade do código de erro específico (400 vs. 409) nesse caso, mas a mensagem ao usuário continua clara.
- **Indicadores básicos**: contadores por situação (aguardando, expirados, convertidos, cancelados) no topo da página "Lista de espera" — calculados no painel a partir da mesma lista já carregada (`GET /api/v1/waitlist` já retorna todos os registros da organização, independente da situação), sem nenhum endpoint novo.
- Sem edição de registro nem conversão/cancelamento em lote — não fazia parte do pedido desta rodada.

Painel: página "Lista de espera" deixou de ser um placeholder — formulário completo (cliente, serviço, período, horário, prioridade, validade), quatro contadores de indicadores e lista com etiquetas de prioridade e situação; cada registro aguardando ganha os botões "Converter em agendamento" (abre um mini-formulário pedindo só o novo início — o fim é calculado a partir da duração do serviço, mesmo padrão da remarcação de agendamentos) e "Cancelar".

Validado com Postgres real: suíte completa do backend (`./mvnw clean verify`), incluindo `ArchitectureTest` (o novo módulo `waitlist` e o novo contrato `AppointmentBooking` não violam limites do Spring Modulith), 16 testes de domínio (`WaitlistEntryTest`, cobrindo transições de status e a lógica de compatibilidade de horário) e 10 testes de integração (`WaitlistControllerTest`, incluindo isolamento multiempresa, ordenação por prioridade e rollback ao converter para um horário já ocupado); suíte do frontend (`ng test`, 42 testes, incluindo os indicadores) e `ng lint`.

## Relacionamento básico (Fatia 5 do roadmap)

Novo módulo `relationships` (já previsto no AGENTS.md, com scaffold vazio criado na Fase 1.1). Cobre os quatro itens do roadmap ("contato pendente; próxima ação; origem; conversão em cliente e agendamento") e os campos do escopo funcional ("origem, última interação, próxima ação e responsável"). Deliberadamente **não** é um CRM completo — sem campanhas, sem funil visual arrastável, sem múltiplos responsáveis por contato — conforme o aviso já registrado no placeholder desta página ("Não transformar esta área em CRM completo ou ferramenta de campanhas").

- **Novo módulo `relationships`** (Flyway `V013`, tabela `relationship_contacts`): nome, telefone, origem, situação, última interação, próxima ação (texto + data) e responsável. Diferente do módulo `clients`, o telefone aqui **não é normalizado nem verificado contra duplicidade** — um contato de relacionamento é um lead, não uma cliente cadastrada; a validação completa de telefone só entra em jogo na conversão, quando um `Client` de verdade é criado. Simplificação deliberada para manter este módulo mais leve, como pedido.
- **Situação**: oito estágios (`NEW_CONTACT`/`IN_SERVICE`/`AWAITING_RESPONSE`/`PENDING_APPOINTMENT`/`SCHEDULED`/`FOLLOW_UP_LATER`/`DID_NOT_SCHEDULE`/`DO_NOT_CONTACT`) — todos os itens do escopo funcional ("novo contato; em atendimento; aguardando resposta; agendamento pendente; agendado; acompanhar depois; não agendou; não contatar"). Ao contrário do ciclo de status do agendamento, aqui a transição entre situações é **livre** (a secretária decide, sem máquina de estados travando a ordem) — é um funil informativo, não um fluxo de negócio com regras rígidas.
- **Responsável**: atribuído automaticamente à usuária que cria o contato (`CurrentActorProvider`), resolvido para nome de exibição via `identity.UserLookup` (mesmo padrão já usado na auditoria). **Sem reatribuição nesta rodada** — o painel ainda não tem uma tela de gestão de equipe/convites para listar outras usuárias da organização, e criar essa tela só para preencher um seletor seria escopo maior que o pedido.
- **Contato pendente**: calculado em tempo de leitura (`isPendingContact`) — verdadeiro quando existe uma "próxima ação" com data já vencida e a situação ainda não é uma das terminais (`SCHEDULED`/`DID_NOT_SCHEDULE`/`DO_NOT_CONTACT`). Mesmo padrão de campo computado (não persistido) já usado no `EXPIRED` da lista de espera.
- **Conversão em cliente e agendamento**: `POST /api/v1/relationships/{id}/convert` (corpo: `serviceId`/`startAt`/`endAt`) faz as duas coisas de uma vez, já que uma sem a outra não faz sentido no fluxo desta ferramenta. Cria a `Client` (se o contato ainda não tiver uma vinculada) via novo contrato público mínimo `clients.ClientRegistration` (implementado por `ClientRegistry`, que já implementa a criação direta) e cria o agendamento via `scheduling.AppointmentBooking` — o mesmo contrato criado para a conversão da lista de espera, reaproveitado sem nenhuma mudança. Bloqueado (`RelationshipContactNotConvertibleException`, 400) se o contato está marcado "não contatar" ou já foi convertido antes. Mesma simplificação de tratamento de erro já registrada na lista de espera: falhas vindas do cadastro de cliente ou do agendamento (telefone inválido, conflito de horário etc.) são exceções internas de outros módulos que este controller não pode importar, então caem num handler genérico de `RuntimeException` mapeado para 409, repassando a mensagem original.
- Sem exportação CSV, sem histórico de interações detalhado (só a última) e sem reatribuição de responsável — fora do escopo desta rodada.

Painel: página "Relacionamento" deixou de ser um placeholder — formulário de cadastro (nome, telefone, origem), dois indicadores (contatos pendentes, total), e lista com etiqueta de origem, situação, responsável, próxima ação e badge de "Contato pendente". Cada contato ganha os botões "Atualizar" (mini-formulário com situação, próxima ação e data) e "Converter em cliente e agendamento" (mini-formulário com serviço e início — fim calculado pela duração do serviço, mesmo padrão já usado na lista de espera e na agenda), este último escondido para contatos já marcados "não contatar" ou já convertidos.

Validado com Postgres real: suíte completa do backend (`./mvnw clean verify`), incluindo `ArchitectureTest` (o novo módulo `relationships` e o novo contrato `clients.ClientRegistration` não violam limites do Spring Modulith), 13 testes de domínio (`RelationshipContactTest`) e 10 testes de integração (`RelationshipControllerTest`, incluindo isolamento multiempresa, contato pendente, bloqueio de conversão para contato "não contatar" e rollback ao converter para um horário já ocupado); suíte do frontend (`ng test`, 47 testes) e `ng lint`.

## Autorização por papel (OWNER/SECRETARY) — fecha o gap de "menor privilégio" (2026-07-23)

Regras definidas explicitamente pelo usuário, aplicadas ao que já existe no código (o que não existe ainda fica documentado como não aplicável por ora, e ganha a restrição automaticamente quando for construído):

- **Somente OWNER**: exportação CSV (as quatro), criar/inativar serviço, alterar horário de funcionamento.
- **OWNER e SECRETARY** (sem mudança): criar/remover bloqueio pontual da agenda — SECRETARY continua com acesso total a clientes, agendamentos (status/cancelamento/remarcação/recorrência), lista de espera e relacionamento.
- **Ainda não existem no código, portanto sem restrição para implementar agora** (ficará OWNER-only automaticamente quando/se forem construídas): editar/reativar serviço, bloqueios recorrentes/férias/multi-dia, gestão de usuárias/papéis/dados da organização/configurações de segurança.

- **`OrganizationAccessGuard`** (novo contrato mínimo em `organizations`, implementado em `organizations.infrastructure` pelo mesmo padrão de `ClientLookup`/`AppointmentOverview`): um único método, `requireOwner()`, que lança `AccessDeniedException` quando `CurrentOrganizationProvider.current().role() != OWNER`. - **Sempre 403, nunca um novo `@ExceptionHandler`**: `AccessDeniedException` já é capturada pelo `ExceptionTranslationFilter` padrão do Spring Security e roteada para o `accessDeniedHandler` já configurado (`SecurityConfig`), o mesmo mecanismo que `SecurityCurrentActorProvider` já usava — backend é a fonte de verdade, a UI só complementa.
- Guard chamado como primeira linha de `ServiceCatalog.create()`/`.deactivate()`, `BusinessHoursManager.replace()` e nos quatro métodos de `ExportService`. `BlockManager` deliberadamente não tocado (continua aberto a OWNER e SECRETARY).

Painel: `AuthService.isOwner` (novo `computed` a partir do papel na organização atual). Botão/link de exportar CSV escondido para quem não é OWNER nas quatro páginas (Clientes, Agenda, Lista de espera, Relacionamento); formulário de criar serviço e botão "Inativar" escondidos na página Serviços (com aviso explicando o motivo); grade de horário de funcionamento e botão "Salvar horários" desabilitados/escondidos na página Configurações — a seção de bloqueios pontuais nessa mesma página continua totalmente aberta, sem nenhuma mudança.

Validado: suíte completa do backend (`./mvnw clean verify`, 208 testes), incluindo `ArchitectureTest` e três novos testes de integração confirmando 403 para SECRETARY em serviços, horário de funcionamento (com bloqueio continuando permitido) e nas quatro exportações; suíte do frontend (`ng test`, 54 testes — os specs pré-existentes de Serviços e Configurações, que testavam ações agora exclusivas de OWNER, foram ajustados para simular uma sessão OWNER via um `AuthService` fake no provider, em vez de remover a nova restrição) e `ng lint`.

**Dois bugs pré-existentes, não relacionados à autorização por papel, encontrados durante o `clean verify` completo desta rodada** (confirmados como preexistentes reproduzindo-os isoladamente contra o código anterior a esta rodada, via `git stash`, antes de corrigir):
- `ExportService`: os quatro métodos de exportação estavam anotados `@Transactional(readOnly = true)` apesar de gravarem um registro de auditoria (`recordExport`) — uma escrita real dentro de uma transação somente leitura. ## Auditoria de segurança (2026-07-23) — correção dos 3 achados de gravidade alta

Um relatório de auditoria de segurança externo, cobrindo autenticação, autorização, isolamento multiempresa, migrações, testes e CI no HEAD da `develope_claude`, apontou 6 achados (3 altos, 2 médios, 1 baixo) e recomendação de **reprovar** integração e produção. Cada achado foi verificado de forma independente lendo o código real antes de agir — todos os 6 se confirmaram. Nesta rodada, corrigidos os 3 de gravidade alta, que a própria auditoria classificou como bloqueadores de integração; os 3 restantes (Actuator exposto a qualquer autenticada, cookie de sessão `Secure` fail-open fora de local, README desatualizado) ficam para uma rodada futura, por decisão explícita do usuário de escopo desta rodada.

- **SEC-001 — fixação de sessão no login**: `SessionAuthenticationService.login()` autenticava e salvava o `SecurityContext` na sessão HTTP existente sem nunca trocar o ID da sessão — uma sessão pré-autenticação plantada por um atacante (ex.: induzindo a vítima a abrir um link que já carrega um `JSESSIONID` do atacante) virava autenticada ao ser reaproveitada no login. Corrigido com um `SessionAuthenticationStrategy` (`ChangeSessionIdAuthenticationStrategy`, composto com a estratégia do SEC-002 abaixo) chamado manualmente logo após autenticar e antes de salvar o contexto — o filtro padrão do Spring Security que faria essa troca automaticamente não participa deste fluxo de login customizado (baseado em `AuthenticationManager.authenticate()` direto, não no filtro `UsernamePasswordAuthenticationFilter`).
- **SEC-002 — reset de senha não revogava sessões ativas**: `PasswordResetService.confirmReset()` só trocava a senha; uma sessão já autenticada antes do reset (por exemplo, de um dispositivo comprometido) continuava válida indefinidamente. Corrigido com um `SessionRegistry` (`SessionRegistryImpl`, em memória) que passa a registrar toda sessão no momento da autenticação (mesma estratégia composta do SEC-001) e um `ConcurrentSessionFilter` na cadeia de segurança que invalida a sessão e responde 401 assim que ela é usada depois de marcada expirada. `confirmReset()` agora chama `SessionInformation.expireNow()` em todas as sessões da usuária pelo e-mail, revogando-as de fato na próxima requisição. **Descoberta lateral durante esta correção, revertida**: a dependência `spring-session-jdbc` e a migração `V002` (tabelas `SPRING_SESSION`/`SPRING_SESSION_ATTRIBUTES`) já existiam, mas `@EnableJdbcHttpSession` nunca foi de fato ativado — ou seja, a sessão sempre foi em memória do container servlet, nunca persistida em banco como a documentação anterior descrevia. Ativar isso agora quebrava em cascata o padrão de todos os testes de integração existentes (que capturam a sessão via `MvcResult.getRequest().getSession(false)`, incompatível com o wrapper de request do Spring Session); decisão explícita de reverter e resolver a revogação de sessão com `SessionRegistry` (em memória, mesmo padrão do `HttpSession` já em uso), deixando a ativação de fato do Spring Session JDBC como um item separado para decisão futura — não fazia parte do escopo aprovado desta rodada.
- **AUTH-003 — SUPPORT com acesso operacional indevido**: `OrganizationAccessGuard` só tinha `requireOwner()`; nenhum código negava SUPPORT em nada, então SUPPORT tinha exatamente o mesmo acesso que SECRETARY em todos os módulos de negócio — contradizendo a regra já definida nesta mesma sessão ("SUPPORT não deve ter acesso operacional permanente") e a própria entrada do PROJECT_STATUS.md que dizia o contrário sem que o código correspondente existisse. Corrigido com um novo método `requireOperator()` (nega só SUPPORT; permite OWNER e SECRETARY), chamado como primeira linha em todo método de aplicação que não já fosse OWNER-only: `ClientRegistry` (create/list), `AppointmentScheduler` (create/createRecurring/reschedule/cancel/list/listByClient/confirm/arrive/start/complete/noShow), `WaitlistService` (create/list/cancel/convert/findCompatible), `RelationshipService` (create/list/update/convert), `BlockManager` (create/list/remove — antes aberto a OWNER+SECRETARY sem restrição nenhuma para SUPPORT), `ServiceCatalog.list()`, `BusinessHoursManager.list()`, `AuditTrailService.recent()`, `DashboardService.today()`. Os métodos que implementam contratos cross-module somente-leitura (`AppointmentOverview`, `WaitlistOverview.findAll`, `RelationshipOverview.findAll`) não receberam guarda própria — já são protegidos indiretamente pela guarda do serviço de nível mais alto que os chama (`ExportService`, já `requireOwner()`; `DashboardService`, agora `requireOperator()`).
  - **Bug revelado pela correção**: `WaitlistController` e `RelationshipController` tinham um `@ExceptionHandler(RuntimeException.class)` genérico (documentado como simplificação deliberada para repassar falhas de conversão de outros módulos) que capturava `AccessDeniedException` antes que ela chegasse ao filtro de segurança, transformando 403 em 409. Corrigido com um `@ExceptionHandler(AccessDeniedException.class)` mais específico que relança a exceção — Spring escolhe o handler mais específico automaticamente, então isso não afeta o handler genérico para as demais exceções.

Painel: nenhuma mudança nesta rodada — o escopo foi inteiramente backend (autenticação e autorização). A UI já esconde ações OWNER-only da rodada anterior; SUPPORT continua sem nenhuma tela própria no painel (nunca teve).

Validado: suíte completa do backend (`./mvnw clean verify`, 218 testes, incluindo `ArchitectureTest`), com testes de regressão novos para os três achados — `AuthControllerTest.rotatesSessionIdOnSuccessfulLogin` (JSESSIONID muda no login), `PasswordResetControllerTest.confirmRevokesActiveSessions` (sessão ativa vira 401 em `/me` logo após o reset) e testes negativos de SUPPORT em `ClientControllerTest`, `AppointmentControllerTest`, `WaitlistControllerTest`, `RelationshipControllerTest`, `AvailabilityControllerTest` (horário de funcionamento e bloqueios), `AuditTrailControllerTest`, `DashboardControllerTest` e `ServiceControllerTest`; suíte do frontend (`ng test`, 54 testes, sem alteração de comportamento).

Validado manualmente contra uma instância real (API + Postgres + Mailpit, via `docker compose` + `./mvnw spring-boot:run -Dspring-boot.run.profiles=local`, com login de verdade e reset de senha real): sem sessão, `/actuator/metrics` e `/actuator/info` respondem 401 (barrados pelo filtro de segurança antes de chegar ao Spring MVC); com sessão autenticada, respondem 404 (o Actuator não registra mais esses endpoints, então nem uma usuária logada os alcança). `/actuator/health` continua 200 em ambos os casos, de propósito — é o único endpoint técnico que precisa ficar público para health check de infraestrutura. Novo teste de integração `ActuatorSecurityTest` (`br.com.agendaplatform.config`) cobre os três casos; `scripts/smoke-test.sh`/`.ps1` ganharam uma checagem confirmando que `/actuator/metrics` nunca mais retorna 200 sem autenticação.
- **CFG-005 — cookie de sessão `Secure` fail-open fora de desenvolvimento**: `SESSION_COOKIE_SECURE` tinha padrão `false`, exigindo que quem fizesse deploy lembrasse de defini-la como `true` — um deploy que esquecesse essa variável enviaria o cookie de sessão também por HTTP. Invertido para fail-closed: o padrão em `application.yml` agora é `true`; só o perfil `local` (`application-local.yml`, desenvolvimento sem HTTPS) sobrescreve para `false`. `docs/operations/deployment-checklist.md` atualizado — a variável deixou de ser uma ação obrigatória de deploy.
- **DOC-006 — README.md desatualizado**: a seção "Ainda não contém" ainda listava lista de espera, relacionamento, relatórios/exportação e recuperação de senha como não implementados, apesar de todos já estarem prontos há várias rodadas — risco real de alguém (humano ou IA) partir de um escopo errado ao revisar ou dar onboarding no projeto. Reescrito para refletir o estado real: seção "Já contém" ampliada com tudo que existe hoje (autorização por papel incluída), seção "Ainda não contém" reduzida ao que de fato falta (calendário visual, edição/reativação de serviço, bloqueios recorrentes, gestão de usuárias pelo painel, acesso técnico de suporte, infraestrutura de produção real).

Painel: nenhuma mudança nesta rodada — as três correções são de configuração/documentação do backend, sem superfície de UI.

Validado: suíte completa do backend (`./mvnw clean verify`, 221 testes, incluindo `ArchitectureTest` e os 3 novos testes do `ActuatorSecurityTest`); validação manual de ponta a ponta contra uma instância real (login, reset de senha com token de verdade via Mailpit, e confirmação do comportamento do Actuator autenticada e não-autenticada) — sem a instabilidade do Claude in Chrome já registrada em rodadas anteriores, já que toda a validação foi feita via curl direto contra a API real.

## Gestão de usuárias pelo painel (fecha uma lacuna do MVP)

Depois de fechar a auditoria de segurança, checagem de completude do MVP (`docs/product/functional-scope.md`) apontou quatro lacunas reais ainda não implementadas (calendário visual, restrição de contato no cliente, intervalo opcional no serviço, gestão de usuárias). O usuário escolheu esta — convite e ativar/desativar secretárias pelo painel — por ser a que mais bloqueava o uso real de um piloto (hoje só existe via SQL direto).

- **Convite**: `POST /api/v1/organizations/members` (só OWNER) cria a usuária com status `INVITED` (valor que já existia no enum `UserStatus` desde a Feature 000, mas nunca tinha sido usado por nenhum código de aplicação) e envia e-mail com link de convite (`/aceitar-convite?token=...`) — mesmo padrão de token de uso único do reset de senha (hash SHA-256, nunca grava o token em texto puro), agora numa tabela própria (`user_invitation_tokens`, Flyway `V015`) e validade maior (7 dias por padrão, configurável via `app.invitation.token-validity-minutes`, já que aceitar um convite de trabalho não tem a mesma urgência de um reset de senha). Convite é sempre para o papel `SECRETARY` — este fluxo não convida outra proprietária nem concede acesso de SUPPORT (definido à parte, conforme já decidido na rodada de autorização por papel).
- **Aceitar convite**: `POST /api/v1/auth/invitations/accept` (público, mesmo padrão de `password-reset/**`) recebe o token e a nova senha, ativa a usuária (`INVITED` → `ACTIVE`) e a partir daí ela já pode logar normalmente.
- **Listar, desativar e reativar**: `GET/POST /api/v1/organizations/members` (só OWNER). Desativar marca o vínculo com a organização como `DISABLED` (não a conta em si — se a mesma pessoa algum dia tiver vínculo com outra organização, ela não é afetada) e **revoga imediatamente todas as sessões ativas** dela, reaproveitando o `SessionRevoker` já criado para o reset de senha (extraído do `PasswordResetService` para `shared.security` nesta rodada, exatamente para ser reaproveitado aqui sem duplicar a lógica). Bloqueado (400) tentar desativar a própria conta ou a proprietária da organização.
- **Status exibido combina dois estados**: o vínculo com a organização (`ACTIVE`/`DISABLED`, já existia) e o status da própria conta (`INVITED`/`ACTIVE`/`DISABLED`) — "convite pendente" aparece enquanto a conta ainda não foi ativada, independente do vínculo (que já nasce `ACTIVE`); depois de desativado o vínculo, o status mostrado é sempre "Desativada", não importa o estado da conta.
- **Decisão de segurança deliberada**: convite é recusado (409, `email_already_registered`) se já existir uma usuária com aquele e-mail em qualquer status ou organização. O modelo de dados (`organization_members`) permite estruturalmente uma mesma usuária ter vínculos com organizações diferentes, mas nenhum outro código do sistema foi construído para suportar isso de verdade (`CurrentOrganizationProvider` assume um único vínculo ativo por usuária e quebraria em runtime se houvesse mais de um) — bloquear e-mail duplicado no convite evita criar esse estado sem suporte, sem precisar reescrever a resolução de organização atual agora.
- **Novo módulo `membership`**: a orquestração do convite/gestão de membros não podia viver dentro de `organizations` nem de `identity` sem criar um ciclo de dependência entre módulos (`identity` já depende de `organizations` desde a Feature 000, então `organizations` não pode depender de volta de `identity`; o mesmo vale para `auditing`, que já depende de ambos). Resolvido com um módulo novo, `membership`, numa camada acima dos três — depende de `identity` (`UserProvisioning`, `UserAccountOverview`, novos contratos), `organizations` (`MembershipRegistry`/`MemberRef`, novo contrato mínimo para criar/listar/desativar/reativar vínculos sem expor `OrganizationMember`/`OrganizationMemberRepository`), `shared.security` (`SessionRevoker`) e `auditing`, sem que nenhum desses módulos precise depender de `membership` de volta. Confirmado pelo `ArchitectureTest` (Spring Modulith), que rejeitou a primeira tentativa (colocar o serviço dentro de `organizations`) por ciclo de dependência antes de chegar a este desenho.
- Auditoria: `MEMBER_INVITED`, `MEMBER_DISABLED`, `MEMBER_REACTIVATED`.

Painel: nova página "Usuárias" (item de navegação só visível para OWNER, mesmo padrão de outras ações restritas), com formulário de convite, lista com etiqueta de papel/situação e botão desativar/reativar por membro (escondido para a proprietária). Nova página pública "Aceitar convite" (acessada pelo link do e-mail, lê o token da query string), no mesmo padrão visual de "Redefinir senha".

Validado: suíte completa do backend (`./mvnw clean verify`, 232 testes, incluindo `ArchitectureTest`), com testes de integração cobrindo convite, e-mail enviado, aceite com token real, negação a SECRETARY, e-mail já cadastrado, desativação com revogação de sessão, reativação, bloqueio de autodesativação e de desativar a proprietária, e isolamento entre organizações; suíte do frontend (`ng test`, 65 testes) e `ng lint`. **Validação manual completa de ponta a ponta contra uma instância real** (API + Postgres + Mailpit, sem nenhum mock, incluindo recriar o volume local do Postgres pelo mesmo motivo já documentado em rodadas anteriores — migração nova abaixo da V900 dos seeds de dev): login como proprietária → convite enviado → e-mail de verdade recebido no Mailpit → aceite com o token real → login da nova secretária funcionando → secretária corretamente barrada (403) da própria tela de gestão de usuárias → proprietária vendo o vínculo como "Ativa" → desativação → sessão da secretária revogada na hora (401 imediato) → reativação → secretária conseguindo logar de novo. A validação visual no navegador (Claude in Chrome) ficou bloqueada pela mesma instabilidade da extensão já registrada em rodadas anteriores ("Browser extension is not connected"); não insistida, dado o nível de confiança já alto pela validação via curl real e pelos testes automatizados de interação de DOM (formulário, botões) nos componentes novos do painel.

## Revisão de qualidade e auditoria de sanidade (pós rodada de segurança + gestão de usuárias)

Pausa de manutenção pedida explicitamente ("reavalie com calma"), depois de várias rodadas seguidas de segurança e funcionalidade nova (autorização por papel, correção dos 6 achados da auditoria, gestão de usuárias). Mesmo escopo de sempre — código morto, duplicação, testes faltantes, magic numbers, módulos grandes — mais uma auditoria de sanidade dedicada (vazamento de segredos, permissões perigosas, path traversal, unwraps inseguros, falta de validação), com a regra de nunca mudar comportamento sem teste cobrindo a mudança.

- **Testes unitários de domínio faltantes, adicionados**: a rodada de gestão de usuárias introduziu lógica de domínio nova (`User.acceptInvitation()`, `UserInvitationToken.markUsed()`, `OrganizationMember.disable()/activate()`) só coberta indiretamente pelos testes de integração — sem teste unitário puro dedicado, diferente do padrão já estabelecido no projeto (ex.: `PasswordResetTokenTest`, `AppointmentTest`). Adicionados `UserTest`, `UserInvitationTokenTest` (espelhando `PasswordResetTokenTest`) e `OrganizationMemberTest` — 9 testes novos, todos passando, nenhuma mudança de comportamento.
- **Código morto**: nenhum encontrado (sem `TODO`/`FIXME`/`XXX`, sem imports não usados nos arquivos novos desta fase, conferido manualmente arquivo por arquivo já que o projeto não tem linter Java configurado).
- **Duplicação**: avaliada e **não mexida** em dois pontos por ser abstração prematura para só 2 usos — `InvitationService`/`PasswordResetService` montam `SimpleMailMessage` com a mesma sequência de 4 linhas (`setTo`/`setSubject`/`setText`/`send`), e `AcceptInvitationPageComponent`/`ConfirmPasswordResetPageComponent` têm estrutura de formulário quase idêntica (token da query string, campo de senha, mensagem de sucesso com redirecionamento). Em ambos os casos, extrair um helper compartilhado acoplaria dois fluxos com mensagens e regras de negócio diferentes por uma economia pequena de linhas — mesmo critério já usado em decisões anteriores de não dividir código só para reduzir contagem de linhas. A duplicação real de lógica (gerar/hashear token) já estava extraída para `SecureTokenGenerator` desde a rodada anterior; a revogação de sessão já estava extraída para `shared.security.SessionRevoker`.
- **Magic numbers**: nenhum encontrado nos módulos novos (validades de token vêm de `@Value` configurável, não de literais soltos).
- **Módulos grandes demais**: reconferido o panorama geral — `AppointmentScheduler` continua o maior arquivo do backend (355 linhas, cresceu com as linhas de guard de autorização), decisão de não dividir já tomada e justificada em rodada anterior, sem motivo novo para reabrir. Os módulos novos desta fase (`membership`, `MembershipService` com 158 linhas) estão bem dentro do razoável.
- **Documentação viva atualizada** (fora do padrão de só mexer no `PROJECT_STATUS.md`): `docs/architecture/security.md` estava desatualizado desde a Feature 000 (ainda dizia "ainda não há cadastro de usuária por API", uma afirmação que virou falsa nesta mesma sessão) — reescrito por completo para refletir autenticação, autorização por papel e a superfície do Actuator como estão hoje, incluindo a limitação conhecida do `SessionRegistry` em memória. `docs/qa/risk-register.md` ganhou uma linha nova para esse mesmo risco (revogação de sessão não se propaga entre instâncias da API).

**Auditoria de sanidade** (segurança, sem achados novos):
- Nenhum segredo hardcoded nos arquivos tocados nesta fase (`.env` fora do controle de versão, confirmado); nenhuma senha/token/chave em texto puro em código ou configuração.
- Nenhuma execução de processo/comando do sistema operacional em código da aplicação (sem `ProcessBuilder`/`Runtime.exec`) — sem risco de comando destrutivo via aplicação.
- Nenhuma manipulação de arquivos com caminho vindo de entrada de usuária — sem superfície de path traversal (a exportação CSV é gerada em memória, nunca lê do disco).
- Nenhum `Optional.get()` sem checagem nem cast inseguro sem `instanceof` nos módulos novos.
- Validação de entrada (`@Valid` + Bean Validation) presente em todos os DTOs novos (`InviteMemberRequest`, `AcceptInvitationRequest`); nenhum limite de tamanho máximo em `displayName`/`email`, mas isso já é o padrão consistente em todo o projeto (nenhum DTO existente limita tamanho de string, ex.: `CreateClientRequest`, `CreateServiceRequest`) — não é uma lacuna introduzida por esta rodada, e corrigir só os DTOs novos criaria inconsistência; fica registrado aqui, não corrigido, por ser uma mudança de escopo maior que afetaria todo o projeto.
- Fluxo de convite responde 409 explícito quando o e-mail já existe (diferente do reset de senha, que é deliberadamente silencioso) — decisão consciente: só uma OWNER já autenticada consegue fazer essa consulta, então o risco de enumeração de e-mails é bem menor do que num endpoint anônimo.

Validado: suíte completa do backend (`./mvnw clean verify`, 241 testes, incluindo `ArchitectureTest` e os 9 testes unitários novos), suíte do frontend (`ng test`, 65 testes, sem alteração) e `ng lint` — nenhuma mudança de comportamento em nenhum teste pré-existente.

## Restrição de contato no cliente (fecha outra lacuna do MVP)

Segunda das quatro lacunas do MVP identificadas na checagem de completude (a primeira, gestão de usuárias, já fechada). Escolhida como a mais recomendada entre as três restantes (calendário visual, restrição de contato, intervalo de serviço) por ser a de menor escopo e risco — não toca a lógica de disponibilidade já grande do `AppointmentScheduler`, diferente do intervalo de serviço.

- **Modelo**: `Client` ganhou `contactRestricted` (boolean) e `contactRestrictionReason` (texto opcional), Flyway `V016`. Interpretação deliberada e mais conservadora do escopo funcional ("restrição de contato"): é uma marcação **administrativa** — a organização não deve mais tomar iniciativa de contatar essa cliente — não um bloqueio técnico de agendamento. Diferente do "não contatar" do módulo `relationships` (que bloqueia a conversão em cliente/agendamento porque ali é um lead ainda não confirmado), aqui a cliente já existe e pode perfeitamente continuar sendo atendida se for ela quem procurar a organização; por isso a restrição não impede criar agendamento para ela — só destaca a situação para quem for decidir se entra em contato ativo.
- **Ações**: `POST /api/v1/clients/{id}/restrict-contact` (motivo opcional) e `POST /api/v1/clients/{id}/lift-contact-restriction`, mesmo nível de acesso das demais ações de cliente (`requireOperator()` — OWNER e SECRETARY, nega só SUPPORT). Auditado (`CLIENT_CONTACT_RESTRICTED`/`CLIENT_CONTACT_RESTRICTION_LIFTED`).

Painel: lista de clientes destaca a linha (borda e fundo diferentes) e mostra a tag "Não contatar" com o motivo, quando houver; botão "Restringir contato" abre um mini-formulário inline pedindo o motivo (opcional), e vira "Remover restrição" enquanto a cliente estiver marcada.

Validado: suíte completa do backend (`./mvnw clean verify`, 248 testes, incluindo `ArchitectureTest`), com 4 testes de domínio novos (`ClientTest`) e 3 de integração (`ClientControllerTest`: fluxo completo com auditoria, negação a SUPPORT, isolamento multiempresa); suíte do frontend (`ng test`, 66 testes), `ng lint` e `ng build`. Validação manual contra uma instância real (Postgres real, recriando o volume local pelo mesmo motivo já documentado — migração nova abaixo dos seeds V900+): criar cliente → restringir contato com motivo → aparece na listagem → remover restrição, tudo via curl direto contra a API.
